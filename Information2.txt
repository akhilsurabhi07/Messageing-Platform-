Project Presentation: Real-Time Communication Platform
This document serves as a comprehensive presentation and final report for the Communication.Plat project. It covers the complete architecture, the technology stack, the unique engineering challenges we solved, and the detailed discussions regarding how the application functions under the hood.

1. Project Overview & Vision
The goal of this project was to build a modern, highly secure, and instant real-time communication platform. Rather than building a traditional, clunky web app, the objective was to engineer a Progressive Web App (PWA) with a premium, glassmorphic dark-mode interface that rivals native applications like Discord or Slack.

The application features:

Secure Two-Factor Authentication (Email OTP)
Real-time Chat Messaging
Peer-to-Peer Video Meetings
Global State Management & Role-Based Access (Admin vs. Employee)
2. The Technology Stack
We strategically selected a modern "Serverless" tech stack to maximize speed and minimize server costs.

Frontend (User Interface)
React (JavaScript): The core framework used to build reusable, dynamic UI components.
Vite: The ultra-fast build tool used for local development (npm run dev) and compiling the final optimized code (npm run build).
React Router: Handles instant, seamless navigation between pages without reloading the browser.
Lucide-React: Provides the sleek, scalable vector icons used throughout the app.
Backend (Serverless Architecture)
Google Firebase: Acts as the Backend-as-a-Service (BaaS). By using the Firebase SDK (firebase.js), the React frontend communicates securely and directly with Google's cloud, completely eliminating the need to write or host a traditional Node.js backend.
Firebase Authentication: Handles password encryption, user sessions, and security.
Firebase Firestore: A NoSQL Document database that stores users, chats, and meetings.
Firebase Hosting: The global Content Delivery Network (CDN) that hosts the live website at commplat-app-akhil.web.app.
External APIs & Integrations
EmailJS: Used to bypass Firebase's default verification links and send custom 6-digit OTP codes directly from the browser.
WebRTC (Native Browser API): Powers the peer-to-peer video meetings.
3. Engineering Challenges & Solutions (Our Discussions)
Throughout the development of this project, we had several deep-dive discussions to solve complex engineering problems. Here is how we tackled them:

Challenge A: The Custom 6-Digit OTP Flow
The Problem: Firebase Authentication's free tier only supports sending long URL links for email verification. You explicitly requested a native 6-digit OTP code that users type directly into the app so they never have to leave the screen. The Solution:

We integrated EmailJS and connected it to your Gmail account.
During Sign Up, we generate a random 6-digit math number, email it silently in the background, and pause the Firebase account creation until the user types the exact matching number.
During Login, we implemented a custom Two-Factor Authentication flow. We verify the user's password first against Firebase. If correct, we freeze their session, send an OTP, and force them to verify the code before allowing access to the dashboard. This prevents hackers from spamming OTPs to random emails.
Challenge B: Building a "Native App" Experience (PWA)
The Problem: You wanted users to be able to download the app directly to their phones and computers without having to go through the Apple App Store or Google Play Store. The Solution:

We used vite-plugin-pwa to turn the website into a Progressive Web App.
The build process automatically generates a manifest.webmanifest (which dictates the app's name, icons, and standalone display mode) and a Service Worker (sw.js).
How updates work: As discussed, when you deploy new code, the Service Worker secretly pings Firebase in the background. It notices the file names have changed (e.g., index-A1.js to index-B2.js), downloads the new files silently, and swaps them out the next time the user closes and reopens the app.
Challenge C: Real-Time Communication vs. Traditional Databases
The Problem: Traditional SQL databases require the app to constantly refresh or "poll" the server every few seconds to ask if there are new messages. This is slow and consumes massive amounts of data. The Solution:

We utilized Firebase's WebSockets via the onSnapshot() command.
This opens a permanent, invisible, live tunnel between the user's phone and the database.
When User A sends a message, they use the addDoc command to toss the message into Firestore. In less than a millisecond, the WebSocket pushes that data directly to User B's screen without User B ever asking for it.
Challenge D: The Video Meeting Infrastructure
The Problem: Building a video meeting platform usually requires expensive servers to route video data from one user to another. The Solution:

We built a custom engine using WebRTC (Web Real-Time Communication).
Instead of routing video through a central server, we use Firestore purely as a "Signaling Server". User A's browser writes an "Offer" to the database, and User B writes an "Answer".
Once they exchange IP addresses through the database, the browsers open a Peer-to-Peer connection. The HD video and audio stream directly between their computers, resulting in zero server bandwidth costs and lightning-fast video.
4. How the Application is Managed
We discussed the workflow of how you (the developer) manage the application from your terminal:

Local Testing (npm run dev): Run inside the Communication.Plat folder. This starts a local server on your machine so you can test code changes instantly in VS Code without affecting the live users.
Compilation (npm run build): Vite compresses thousands of lines of React code into a few highly optimized, minified files placed in a dist/ folder.
Deployment (firebase deploy): This command takes the compiled dist/ folder and uploads it to Google's global servers, instantly updating your live application.
Database Management: Data is entered using setDoc (when we want to name the folder exactly after the User's ID) and addDoc (when we want Firebase to generate a random, unique ID for things like chat messages).
Conclusion
By combining React, Firebase, WebRTC, and EmailJS, we successfully engineered a highly scalable, serverless communication platform. The application is capable of handling millions of concurrent users, streams real-time data seamlessly, and provides a premium, installable user experience across all devices.

1. The Two-Factor OTP System (EmailJS + React)
To bypass Firebase's default email links, we built a custom flow. Here is the exact code architecture for how we securely log a user in with a 6-digit OTP.

The EmailJS Utility (src/utils/emailjs.js)
First, we created a utility function that talks to the EmailJS API securely.

javascript

import emailjs from '@emailjs/browser';
const SERVICE_ID = 'service_pkfwzf6';
const TEMPLATE_ID = 'template_vogmagi';
const PUBLIC_KEY = 'swg7ncndMcO1VVHN7';
emailjs.init(PUBLIC_KEY);
export const sendOtpEmail = async (to_email, otpCode) => {
  const templateParams = {
    email: to_email,
    passcode: otpCode,
  };
  return emailjs.send(SERVICE_ID, TEMPLATE_ID, templateParams);
};
The Login Coordination (src/pages/Login.jsx)
In the Login interface, we don't just log the user in immediately. We verify their password secretly, freeze the UI, and send the email.

javascript

const handleSubmit = async (e) => {
    e.preventDefault();
    setLoading(true);
    try {
      // 1. Verify password first by attempting a secret login
      await login(email, password);
      
      // 2. Immediately log them out to freeze their session
      await logout();
      
      // 3. Generate a random 6-digit math number
      const otp = Math.floor(100000 + Math.random() * 900000).toString();
      setGeneratedOtp(otp);
      
      // 4. Send the OTP via the EmailJS utility we built above
      await sendOtpEmail(email, otp);
      
      // 5. Flip the React State to show the OTP input screen
      setOtpStep(true);
    } catch (err) {
      setError('Invalid credentials');
    }
};
2. Real-Time WebSockets (Firestore onSnapshot)
Traditional apps use REST APIs (HTTP GET requests) to pull data, which requires reloading the page. We used Firebase's onSnapshot to open a WebSocket—a live, invisible tunnel to the database.

Here is the code from the meeting/chat rooms that makes messages appear instantly without refreshing:

javascript

import { collection, query, orderBy, onSnapshot } from 'firebase/firestore';
useEffect(() => {
    // 1. Point to the specific meeting's message folder
    const q = query(
        collection(db, `meetings/${meetingId}/messages`), 
        orderBy('timestamp', 'asc')
    );
    // 2. onSnapshot opens the WebSocket tunnel. 
    // This function automatically runs EVERY TIME a new message is added to the database.
    const unsubscribe = onSnapshot(q, (snapshot) => {
      
      // 3. Extract the new data
      const msgs = snapshot.docs.map(d => ({ id: d.id, ...d.data() }));
      
      // 4. Update the React state, causing the screen to instantly re-render with the new text
      setMessages(msgs);
    });
    // 5. Close the tunnel when the user leaves the page to save battery/bandwidth
    return () => unsubscribe();
}, [meetingId]);
3. WebRTC: Peer-to-Peer Video Meetings
The most complex part of the app is the video meeting engine. It does not use external video servers (like Zoom does). Instead, it uses WebRTC to stream video directly from User A's computer to User B's computer.

Step 1: Getting Camera Access
We use the browser's native navigator.mediaDevices API to turn on the webcam and microphone.

javascript

const requestMedia = async () => {
    // Ask the user for permission to use their Camera and Mic
    const stream = await navigator.mediaDevices.getUserMedia({
        video: { facingMode: 'user' },
        audio: true
    });
    
    // Save the raw video data into React State so we can display it in an HTML <video> tag
    setLocalStream(stream);
};
Step 2: The WebRTC Peer Connection
To connect two computers, we use the RTCPeerConnection API. We provide it with Google's free "STUN" servers, which act like a phonebook to help the two computers find each other's IP addresses on the internet.

javascript

const ICE_SERVERS = {
  iceServers: [
    { urls: 'stun:stun.l.google.com:19302' },
    { urls: 'stun:stun1.l.google.com:19302' }
  ]
};
// Create the connection tunnel
const pc = new RTCPeerConnection(ICE_SERVERS);
// Take the camera video stream from Step 1 and shove it into the tunnel
streamRef.current.getTracks().forEach(track => {
    pc.addTrack(track, streamRef.current)
});
// When video pops out the other side of the tunnel from User B, save it to state
pc.ontrack = (event) => {
    if (event.streams && event.streams[0]) {
        setRemoteStreams(prev => ({ ...prev, [targetUid]: event.streams[0] }));
    }
};
Step 3: Signaling via Firestore
For the two computers to open that tunnel, they have to exchange security keys (called Offers and Answers). Since they can't talk directly yet, they leave notes for each other in the Firestore database!

javascript

// User A creates an "Offer" and writes it to the database
const offer = await pc.createOffer();
await pc.setLocalDescription(offer);
await addDoc(collection(db, `meetings/${meetingId}/signals`), {
    from: userData.uid,
    to: targetUid,
    type: 'offer',
    data: { type: offer.type, sdp: offer.sdp }
});
User B's React code is listening to that database collection using onSnapshot. When it sees User A's Offer, it reads it, creates an "Answer", and writes it back to the database. Once they exchange this handshake, the Peer-to-Peer video tunnel opens, and the database steps out of the way!
